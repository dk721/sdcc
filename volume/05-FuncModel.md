# Лабораторная работа №5  
**Тема:** Проектирование функциональной модели в SADT/IDEF0 для системы управления доступом и учёта действий пользователей

## Задача 1. Контекстная диаграмма (уровень A-0)

Система рассматривается как единая функция:

![IDEF0](../include/idef.svg)

**Наименование блока:**  
A0. Управлять доступом и учётом действий пользователей в информационной системе.

**Стрелки контекстной диаграммы (ICOM):**

| Тип стрелки | Наименование на диаграмме | Смысловое содержание |
|-------------|---------------------------|----------------------|
| **Входы (Input)** | Запросы доступа | Учётные данные пользователя, запросы к защищённым ресурсам |
| | Кадровые данные | Сведения о сотрудниках, служащие основой для учётных записей |
| **Выходы (Output)** | Авторизованный доступ / Отказ | Результат обработки запроса: успешное предоставление доступа или блокировка |
| | Журнал аудита | Защищённый массив зарегистрированных событий безопасности |
| | Отчёты и оповещения | Аналитические сводки, алерты об инцидентах для администраторов |
| **Управления (Control)** | Политики безопасности | Корпоративные правила парольной защиты, блокировок, ролевая модель |
| | Нормативные требования | Стандарты (ISO 27001, GDPR), внутренние регламенты хранения логов |
| **Механизмы (Mechanism)** | Платформа IAM | Программно-аппаратный комплекс: сервера, агенты, базы данных |
| | Администратор | Уполномоченный сотрудник, настраивающий и контролирующий систему |

Графически на диаграмме в центре располагается прямоугольник «A0», к левой стороне подходят две стрелки входов, к правой — три стрелки выходов, к верхней — две стрелки управления, к нижней — две стрелки механизмов. Все объекты внешней среды вынесены за пределы блока.

## Задача 2. Декомпозиция функций до третьего уровня

### Диаграмма A0 (первый уровень)

Функция A0 разбивается на пять дочерних блоков:

- **A1.** Идентифицировать и аутентифицировать пользователя  
- **A2.** Авторизовать доступ к ресурсу  
- **A3.** Вести аудит событий  
- **A4.** Мониторить и оповещать  
- **A5.** Администрировать систему  

На этой диаграмме сохраняются все граничные стрелки контекстного уровня. Дополнительно вводятся внутренние потоки данных:

- Выход A1 «Аутентифицированный субъект» становится входом для A2.
- Выход A2 «Факты доступа» передаётся в A3.
- A3 формирует «Структурированный поток событий» для A4.
- А5 вырабатывает «Настроенные политики» (управление для A2) и «Учётные записи от администрирования» (механизм для A1).

Все блоки используют общий механизм «Платформа IAM». Блок A5 дополнительно требует механизм «Администратор».

### Уровень 2

Декомпозируются наиболее содержательные функции.

**Декомпозиция A2 «Авторизовать доступ к ресурсу»:**  
Блок A2 состоит из двух подфункций:

- **A21.** Проверить права субъекта  
- **A22.** Применить решение о доступе  

A21 получает «Аутентифицированного субъекта» и «Характеристики ресурса», вырабатывает «Вердикт», который передаётся в A22. A22, управляемый «Политиками безопасности» и исполняемый механизмом «Точка применения решения (PEP)», создаёт выходы «Авторизованный доступ / Отказ» и «Факты доступа».

**Декомпозиция A3 «Вести аудит событий»:**  
Включает три функции:

- **A31.** Собрать и буферизовать события  
- **A32.** Нормализовать и обогатить событие  
- **A33.** Сохранить в защищённый журнал  

Поток данных: A31 получает «Факты доступа» → выход «Буфер сырых событий» → A32 → выход «Обогащённая структурированная запись» → A33. Функция A33 создаёт «Журнал аудита» и «Структурированный поток событий» для A4.

**Декомпозиция A4 «Мониторить и оповещать»:**  
- **A41.** Коррелировать события  
- **A42.** Генерировать и отправлять оповещения  

A41 из «Структурированного потока событий» формирует «Выявленные инциденты», которые передаются в A42. Выход A42 — «Отчёты и оповещения».

**Декомпозиция A5 «Администрировать систему»:**  
- **A51.** Управлять учётными записями  
- **A52.** Управлять ролями и политиками  
- **A53.** Формировать отчёты соответствия  

A51 получает «Кадровые данные» и выдаёт «Учётные записи от администрирования». A52 производит «Настроенные политики». A53 — «Отчёты и оповещения» (по требованиям регуляторов).

### Уровень 3

Детализируется функция A21 «Проверить права субъекта»:

- **A211.** Загрузить роли пользователя  
  *Вход:* «Аутентифицированный субъект», *Выход:* «Список ролей», *Механизм:* «Хранилище учётных записей».

- **A212.** Получить разрешения ролей  
  *Вход:* «Список ролей», *Выход:* «Набор разрешений», *Механизм:* «База политик», *Управление:* «Политики доступа».

- **A213.** Сопоставить операцию с разрешениями  
  *Вход:* «Набор разрешений» и «Характеристики ресурса», *Выход:* «Вердикт», *Управление:* «Политики доступа», *Механизм:* «Точка принятия решений (PDP)».

Диаграмма A21 показывает строгую последовательность обработки: от получения ролей до вынесения вердикта.

## Задача 4. Анализ типов связности между функциями

В построенной иерархии выделены следующие типы связей.

1. **Последовательная связность по данным (output-to-input).**  
   Основной поток запроса проходит через функции:
   - A1 → A2 (Аутентифицированный субъект),
   - A2 → A3 (Факты доступа),
   - A3 → A4 (Структурированный поток событий).  
   Каждый последующий блок не может начать работу без результата предыдущего. Этот тип связности гарантирует сквозную обработку каждого запроса.

2. **Обратная связь по управлению.**  
   Функция A5 вырабатывает «Настроенные политики», которые служат управлением для A2 (авторизации). Если администратор изменяет ролевую модель, это немедленно сказывается на правилах авторизации. Аналогично, «Учётные записи от администрирования» через механизм A1 влияют на допустимость аутентификации. Таким образом, административный контур замыкается через управление.

3. **Параллельная (асинхронная) связность.**  
   Функции A3 (аудит) и A4 (мониторинг) работают в постоянном фоновом режиме: A3 непрерывно пишет журнал, A4 независимо читает поток событий и анализирует. Временной разрыв между записью и анализом не нарушает работу системы. Администрирование (A5) также выполняется параллельно с обработкой запросов доступа.

4. **Связность по общим механизмам.**  
   Блоки A1, A2, A3, A4 опираются на единый механизм «Платформа IAM». Отказ этого механизма (например, выход из строя сервера аутентификации) приводит к невозможности выполнения всех четырёх функций. Это создаёт единую точку отказа, что должно учитываться при проектировании отказоустойчивости.

5. **Связность по управлению от внешних сущностей.**  
   Контекстные стрелки «Политики безопасности» и «Нормативные требования» пронизывают всю иерархию и появляются на многих дочерних диаграммах (A1, A2, A3, A4, A5). Это отражает сквозное действие корпоративных и законодательных норм на все аспекты системы.

Все переходы между уровнями сбалансированы: суммарные входы/выходы дочерних диаграмм строго соответствуют входам/выходам родительского блока.
